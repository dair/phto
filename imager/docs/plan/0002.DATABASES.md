# 0002 — Multiple Databases (One Per Output Root)

## Goal

Each storage root gets its own SQLite database. All databases are kept identical — every write operation is applied to all databases atomically (all-or-nothing). This mirrors the existing multi-root file storage semantics: if any database write fails, all are rolled back.

All database operations in `MultiDatabase` are parallelized using C++23 coroutines. Each per-DB operation runs as a coroutine dispatched to a thread pool, and the caller awaits all of them concurrently. This eliminates the N× sequential latency penalty.

## Configuration changes

### New TOML format

The current flat config:

```toml
[storage]
roots = ["/mnt/disk1/images", "/mnt/disk2/images-backup"]

[database]

```

Becomes an array of output targets, each pairing a root with a database:

```toml

root = "/mnt/disk1/images"
database = "/mnt/disk1/imager.db"

[[targets]]
root = "/mnt/disk2/images-backup"
database = "/mnt/disk2/imager.db"
```

- `[[targets]]` is a TOML array of tables. Each entry has exactly two required keys: `root` (storage directory path) and `database` (SQLite file path).
- At least one target is required. The parser throws on empty/missing `[[targets]]`.
- The old `[storage]` and `[database]` sections are removed entirely — no backward compatibility shim.

## Coroutine infrastructure

### Why coroutines

With N databases, sequential writes cost N× the single-DB latency. C++23 coroutines let us dispatch each per-DB operation concurrently and await all results, reducing wall-clock time to ~1× (the slowest DB) instead of N×. The same mechanism is used for parallel reads when needed (e.g., future consistency checks).

The project already uses C++23 (`CMAKE_CXX_STANDARD 23`) and Clang 19, which has full `<coroutine>` support.

### No external coroutine library

We implement a minimal, self-contained coroutine infrastructure. No external libraries (cppcoro, libunifex, etc.) are introduced. The primitives are small enough to own directly:

- `Task<T>` — a lazy coroutine return type (the coroutine does not start until awaited)
- `ThreadPool` — a simple thread pool that coroutines can `co_await` to resume on a worker thread
- `whenAll(vector<Task<T>>)` — awaits all tasks concurrently, collects results or exceptions

These live in new files under `imager/src/coro/`.

### `Task<T>` (`imager/src/coro/Task.h`)

A lazy, move-only coroutine return type. Conforms to the C++23 coroutine promise/awaiter protocol.

```cpp
#pragma once
#include <coroutine>
#include <exception>
#include <optional>
#include <utility>

namespace imager::coro {

template <typename T = void>
class Task {
public:
    struct promise_type;
    using handle_type = std::coroutine_handle<promise_type>;

    struct promise_type {
        std::optional<T>       value;
        std::exception_ptr     exception;
        std::coroutine_handle<> continuation;

        Task get_return_object() {
            return Task{handle_type::from_promise(*this)};
        }
        std::suspend_always initial_suspend() noexcept { return {}; } // lazy
        auto final_suspend() noexcept {
            struct Awaiter {
                bool await_ready() noexcept { return false; }
                std::coroutine_handle<> await_suspend(handle_type h) noexcept {
                    if (h.promise().continuation)
                        return h.promise().continuation;
                    return std::noop_coroutine();
                }
                void await_resume() noexcept {}
            };
            return Awaiter{};
        }
        void return_value(T v) { value = std::move(v); }
        void unhandled_exception() { exception = std::current_exception(); }
    };

    // Awaiter: when someone co_awaits this Task
    bool await_ready() const noexcept { return false; }
    std::coroutine_handle<> await_suspend(std::coroutine_handle<> caller) {
        m_handle.promise().continuation = caller;
        return m_handle; // start the lazy coroutine
    }
    T await_resume() {
        if (m_handle.promise().exception)
            std::rethrow_exception(m_handle.promise().exception);
        return std::move(*m_handle.promise().value);
    }

    explicit Task(handle_type h) : m_handle(h) {}
    ~Task() { if (m_handle) m_handle.destroy(); }
    Task(Task&& o) noexcept : m_handle(std::exchange(o.m_handle, nullptr)) {}
    Task& operator=(Task&&) = delete;
    Task(const Task&) = delete;

private:
    handle_type m_handle;
};

// Specialization for Task<void>
template <>
class Task<void> { /* analogous, with return_void() instead of return_value() */ };

} // namespace imager::coro
```

The `Task<void>` specialization follows the same pattern but uses `return_void()` and stores no value. Full implementation in the source file.

### `ThreadPool` (`imager/src/coro/ThreadPool.h`)

A fixed-size thread pool with an awaitable `schedule()` method. When a coroutine `co_await`s `pool.schedule()`, it suspends and resumes on one of the pool's worker threads.

```cpp
#pragma once
#include <coroutine>
#include <condition_variable>
#include <deque>
#include <mutex>
#include <thread>
#include <vector>

namespace imager::coro {

class ThreadPool {
public:
    explicit ThreadPool(size_t numThreads);
    ~ThreadPool(); // joins all threads

    // Awaitable: co_await pool.schedule() resumes the coroutine on a worker thread
    auto schedule() {
        struct Awaiter {
            ThreadPool& pool;
            bool await_ready() const noexcept { return false; }
            void await_suspend(std::coroutine_handle<> h) {
                pool.enqueue(h);
            }
            void await_resume() noexcept {}
        };
        return Awaiter{*this};
    }

private:
    void enqueue(std::coroutine_handle<> h);
    void workerLoop();

    std::vector<std::jthread>              m_workers;
    std::deque<std::coroutine_handle<>>    m_queue;
    std::mutex                             m_mutex;
    std::condition_variable                m_cv;
    bool                                   m_stop{false};
};

} // namespace imager::coro
```

The pool size defaults to `std::thread::hardware_concurrency()` but is capped at a reasonable minimum (e.g., 2) since we typically have 2–4 databases. Created once in `MultiDatabase` and shared across all operations.

### `whenAll` (`imager/src/coro/WhenAll.h`)

Awaits a vector of `Task<T>` concurrently. Returns a `vector<T>` of results (preserving order). If any task throws, the first exception is rethrown after all tasks complete.

```cpp
namespace imager::coro {

/// Await all tasks concurrently. Returns results in order.
/// If any task threw, rethrows the first exception after all complete.
template <typename T>
Task<std::vector<T>> whenAll(std::vector<Task<T>> tasks);

/// Specialization for Task<void>: just awaits all, rethrows first exception.
Task<void> whenAll(std::vector<Task<void>> tasks);

} // namespace imager::coro
```

Implementation approach: `whenAll` starts all tasks, uses an atomic counter to track completions, and resumes the caller when the counter reaches N. Each task's result (or exception) is stored in a pre-allocated vector slot indexed by task position.

### Synchronous bridge: `blockOn`

Since the facade's public API is synchronous, we need a way to run a coroutine to completion from a non-coroutine context. A simple `blockOn(Task<T>)` helper starts the coroutine on the thread pool and blocks the calling thread until it completes.

```cpp
namespace imager::coro {

/// Run a coroutine to completion, blocking the calling thread.
/// Returns the coroutine's result or rethrows its exception.
template <typename T>
T blockOn(ThreadPool& pool, Task<T> task);

void blockOn(ThreadPool& pool, Task<void> task);

} // namespace imager::coro
```

Implementation: wraps the task in a coroutine that signals a `std::binary_semaphore` (or `std::promise`) on completion, then waits on it.

## Database layer changes

### New class: `MultiDatabase`

**Location**: `imager/src/` as an internal implementation detail.

File: `imager/src/MultiDatabase.h` and `imager/src/MultiDatabase.cpp`.

```cpp
namespace imager {

class MultiDatabase {
public:
    /// Opens/creates one db::Database per target.
    /// Creates the internal thread pool.
    explicit MultiDatabase(const std::vector<config::TargetConfig>& targets);
    ~MultiDatabase();

    MultiDatabase(const MultiDatabase&) = delete;
    MultiDatabase& operator=(const MultiDatabase&) = delete;

    // --- Forwarded write operations (all-or-nothing, parallel) ---
    void addFile(const std::string& id, const std::string& name,
                 uint64_t size, const std::string& ext);
    void deleteFile(const std::string& id);
    void editFileName(const std::string& id, const std::string& newName);
    void addTag(const std::string& name);
    void deleteTag(const std::string& name);
    void bindTag(const std::string& fileId, const std::string& tagName);
    void unbindTag(const std::string& fileId, const std::string& tagName);

    // --- Read operations (from first DB) ---
    std::optional<db::File> getFile(const std::string& id);
    std::vector<db::File> getAllFiles(std::optional<db::Pagination> page = std::nullopt);
    bool fileExists(const std::string& id);
    uint64_t fileCount();
    std::vector<std::string> getAllTags(std::optional<db::Pagination> page = std::nullopt);
    bool tagExists(const std::string& name);
    uint64_t tagCount();
    std::vector<std::string> getTagsForFile(const std::string& fileId,
                                             std::optional<db::Pagination> page = std::nullopt);
    std::vector<db::File> getFilesByTags(const std::vector<std::string>& tagNames,
                                          std::optional<db::Pagination> page = std::nullopt);

private:
    std::vector<std::unique_ptr<db::Database>> m_dbs;
    coro::ThreadPool                           m_pool;

    /// Helper: dispatch an operation to all DBs in parallel, await all.
    /// Op is a callable: void(db::Database&).
    /// On any failure, runs compensate(db::Database&, size_t index) on
    /// succeeded DBs, then rethrows.
    template <typename Op, typename Compensate>
    void parallelWriteAll(Op&& op, Compensate&& compensate);

    /// Simpler overload for non-compensatable operations
    template <typename Op>
    void parallelWriteAll(Op&& op);
};

} // namespace imager
```

### Parallel write pattern (coroutine-based)

The core helper `parallelWriteAll` works as follows:

```cpp
template <typename Op, typename Compensate>
void MultiDatabase::parallelWriteAll(Op&& op, Compensate&& compensate) {
    // Build one Task<void> per database
    auto makeTasks = [&]() -> std::vector<coro::Task<void>> {
        std::vector<coro::Task<void>> tasks;
        tasks.reserve(m_dbs.size());
        for (auto& db : m_dbs) {
            tasks.push_back([](coro::ThreadPool& pool, db::Database& db, Op& op)
                -> coro::Task<void> {
                co_await pool.schedule();  // resume on worker thread
                op(db);                     // execute the DB operation
            }(m_pool, *db, op));
        }
        return tasks;
    };

    // Await all tasks; whenAll collects all exceptions
    auto allTask = [&]() -> coro::Task<std::vector<std::optional<std::exception_ptr>>> {
        // ... launch all, track per-task success/failure
    };

    auto results = coro::blockOn(m_pool, /* aggregate task */);

    // If any failed, compensate the ones that succeeded
    // Then rethrow the first exception
}
```

**Example: `addFile` implementation**:

```cpp
void MultiDatabase::addFile(const std::string& id, const std::string& name,
                             uint64_t size, const std::string& ext) {
    parallelWriteAll(
        // operation
        [&](db::Database& db) { db.addFile(id, name, size, ext); },
        // compensation
        [&](db::Database& db, size_t) { db.deleteFile(id); }
    );
}
```

Each per-DB `addFile` call runs as a coroutine on a thread pool worker. All N calls execute concurrently. If DB #2 fails while DB #0 and #1 succeed, compensation (`deleteFile`) is dispatched in parallel to DB #0 and #1.

### Write operation semantics (all-or-nothing, parallel)

For every write method:

1. Build N coroutines (one per DB), each calling `co_await pool.schedule()` then executing the operation.
2. Await all N with `whenAll`. Each task stores its own exception if it fails.
3. After all complete, check for failures:
   - If all succeeded → done.
   - If any failed → dispatch compensation coroutines in parallel to the succeeded DBs, await those, then rethrow the first original exception.

Compensation operations per write:
- `addFile` → `deleteFile` on succeeded DBs.
- `deleteFile` → re-insert captured `File` record + re-bind captured tags on succeeded DBs.
- `addTag` → `deleteTag` on succeeded DBs.
- `deleteTag` → re-insert tag + re-bind captured file associations on succeeded DBs.
- `bindTag` → `unbindTag` on succeeded DBs.
- `unbindTag` → `bindTag` on succeeded DBs.
- `editFileName` → `editFileName(id, originalName)` on succeeded DBs (name captured before dispatch).

**Rollback strategy for destructive operations** (`deleteFile`, `deleteTag`):

For delete operations, the safe pattern is:
1. Before dispatching coroutines, read the record from the first DB (synchronous, single read) to capture current state.
2. For `deleteFile`: capture the `File` record and its tag bindings.
3. For `deleteTag`: capture the tag name and all file bindings.
4. Dispatch delete coroutines in parallel. On partial failure, re-insert captured data into already-deleted DBs (also in parallel).

If compensation itself fails (double fault), throw a `DatabaseException` with a clear message indicating inconsistent state.

### Read operation semantics

All reads go to `m_dbs[0]` (the first database). Since all databases are kept identical, any would work. Reads are **not** dispatched to the thread pool — they run synchronously on the caller's thread, same as before. There is no benefit to parallelizing a single-DB read.

## Open questions / risks

- **Rollback complexity**: compensating `deleteFile` and `deleteTag` requires capturing full state before the parallel dispatch. This adds one synchronous read before each destructive write. Acceptable overhead for correctness.
- **Thread pool sizing**: the pool is shared across all `MultiDatabase` operations. With 2–4 databases, a pool of 4 threads is sufficient. Over-subscribing is harmless since DB operations are I/O-bound, not CPU-bound.
- **Coroutine overhead**: the `Task<T>` / `whenAll` / `blockOn` machinery adds one heap allocation per coroutine frame (standard C++ coroutine behavior). For the expected 2–4 databases per operation, this is negligible.
- **No external coroutine library**: we intentionally avoid cppcoro, libunifex, etc. to keep dependencies minimal. The coroutine primitives are ~200 lines total and fully self-contained.
- **`db::Database` thread safety**: each `db::Database` instance already uses `std::shared_mutex` internally (per `database/CLAUDE.md`). Since each coroutine operates on a *different* `db::Database` instance, there is no contention between parallel coroutines within a single `parallelWriteAll` call. The facade's `writeMutex` still serializes top-level write operations.
- **Consistency verification**: there is no mechanism to detect or repair drift between databases after the fact (e.g., if a process crashes mid-write). A future enhancement could add a consistency-check / repair command. This is out of scope here.
