# 0003 — Whole-Library Coroutine Parallelization

## Goal

Refactor the entire `libimager` facade to use C++23 coroutines for all operations that can benefit from parallelism. This builds on the coroutine infrastructure introduced in 0002 (Task, ThreadPool, whenAll, blockOn) and extends it beyond MultiDatabase to cover file storage, the addImage pipeline, and query fan-out.

## Prerequisites

- 0002 (multiple databases + coroutine infrastructure) is implemented first.
- Clang 19 + C++23 are already in use.
- The coroutine primitives (`Task<T>`, `ThreadPool`, `whenAll`, `blockOn`) from 0002 exist in `imager/src/coro/`.

## Parallelism opportunities

The current `Imager` implementation is fully sequential. The following operations have independent sub-tasks that can run concurrently:

| Operation | Current behavior | Parallelism opportunity |
|-----------|-----------------|------------------------|
| `addImage` steps 2+3 | Validate, then hash (sequential) | Validate and hash in parallel (both are read-only on the same buffer) |
| `addImage` step 5 | `FileStorage::writeFile` iterates roots sequentially | Write to all roots in parallel |
| `addImage` step 6 | `MultiDatabase::addFile` (already parallel from 0002) | Already parallel |
| `deleteImage` storage | `FileStorage::deleteFile` iterates roots sequentially | Delete from all roots in parallel |
| `getImagesByTags` | N+1: fetches tags per file in a loop | Fan out all `getTagsForFile` calls in parallel |
| `listImages` | N+1: fetches tags per file in a loop | Fan out all `getTagsForFile` calls in parallel |
| `getImage` | One DB read + one tag fetch (sequential) | Two calls could overlap, but benefit is negligible for a single file — leave sequential |
| `getImageData` | DB read then storage read | Sequential dependency — leave as-is |

### What NOT to parallelize

- **Single-DB reads** (`getFile`, `fileExists`, `fileCount`, etc.): these are single SQLite queries against `m_dbs[0]`. Adding coroutine overhead for a single call hurts rather than helps.
- **Validation itself**: each validator (JPEG, PNG) is a single call to libjpeg/libpng. These are CPU-bound but each processes the data once — no sub-task parallelism within them.
- **Config loading**: runs once at startup, not performance-sensitive.
- **`getImage`**: only 2 sequential calls for a single file — coroutine setup cost would dominate.
- **`getImageData`**: the DB read and storage read have a data dependency (need `ext` from DB to find file on disk).

## Detailed design

### 0. New type: `imager::Blob` — shared-ownership data buffer

The current `addImage` signature takes a raw `const uint8_t* data, size_t size`. When coroutines are dispatched to pool threads, the caller must guarantee the pointer stays valid until all coroutines complete. Today this works because `blockOn` blocks the caller, but it's fragile — any future change that makes `addImage` non-blocking (e.g., Phase 2 HTTP layer queuing work) would introduce a use-after-free.

Replace the raw pointer+size pair with a value type that carries shared ownership of the underlying buffer:

#### File location

`Blob` lives in its own header in a new `types/` subdirectory, separate from the existing `Types.h` which holds `ErrorCode`, `ImageInfo`, and `AddResult`:

```
imager/
├── include/imager/
│   ├── Types.h              # ErrorCode, ImageInfo, AddResult (unchanged)
│   └── types/
│       └── Blob.h           # Blob class
```

`Types.h` gains `#include "types/Blob.h"` so existing consumers that include `Types.h` still see `Blob`.

#### Definition (`imager/include/imager/types/Blob.h`)

```cpp
#pragma once

#include <cstdint>
#include <cstring>
#include <memory>
#include <vector>

namespace imager {

/// Shared-ownership binary buffer.
///
/// Two-phase construction for reading:
///   Blob blob(fileSize);             // allocates writable buffer
///   stream.read(blob.writableData(), blob.size());  // fill it
///   blob.freeze();                   // make immutable, enable sharing
///
/// Adoption construction for existing data:
///   Blob blob = Blob::fromVector(std::move(vec));  // takes ownership
///
/// Once frozen, cheap to copy (shared_ptr refcount bump).
/// Safe to pass into coroutines dispatched to a thread pool.
class Blob {
public:
    /// Construct an empty blob.
    Blob() = default;

    /// Construct a writable blob of the given size.
    /// Allocates `size` bytes on the heap, ready to be filled via writableData().
    /// The blob starts in a mutable state; call freeze() when done writing.
    explicit Blob(size_t size)
        : m_data(size > 0 ? std::shared_ptr<uint8_t[]>(new uint8_t[size])
                          : nullptr)
        , m_size(size) {}

    /// Construct by taking ownership of an existing shared_ptr.
    Blob(std::shared_ptr<uint8_t[]> data, size_t size)
        : m_data(std::move(data)), m_size(size), m_frozen(true) {}

    /// Construct by adopting a vector's contents.
    static Blob fromVector(std::vector<uint8_t>&& vec) {
        auto size = vec.size();
        if (size == 0) return {};
        auto* raw = new uint8_t[size];
        std::memcpy(raw, vec.data(), size);
        Blob b;
        b.m_data = std::shared_ptr<uint8_t[]>(raw);
        b.m_size = size;
        b.m_frozen = true;
        return b;
    }

    /// Pointer to the data for reading. Valid in any state.
    const uint8_t* data() const noexcept { return m_data.get(); }

    /// Writable pointer for filling the buffer.
    /// Only valid before freeze() is called.
    uint8_t* writableData() noexcept { return m_data.get(); }

    /// Mark the blob as immutable. After this call, writableData() must not
    /// be used. The blob becomes safe to copy and share across threads.
    void freeze() noexcept { m_frozen = true; }

    /// Whether the blob has been frozen (immutable).
    bool frozen() const noexcept { return m_frozen; }

    size_t size() const noexcept { return m_size; }
    bool   empty() const noexcept { return m_size == 0; }

private:
    std::shared_ptr<uint8_t[]> m_data;
    size_t                     m_size{0};
    bool                       m_frozen{false};
};

} // namespace imager
```

Key properties:
- **Two-phase construction**: `Blob(size)` allocates a writable buffer. The caller fills it via `writableData()`, then calls `freeze()`. This avoids a copy when reading files from disk — read directly into the blob's buffer.
- **Adoption construction**: `Blob::fromVector(vec&&)` takes ownership of existing data, already frozen.
- **No copy-from-pointer constructor**: eliminated. Callers who have a raw `uint8_t*` should construct a `Blob(size)`, `memcpy` into `writableData()`, and `freeze()`. This makes the copy explicit at the call site rather than hidden in a constructor.
- **Cheap to copy once frozen**: copying a `Blob` is a `shared_ptr` refcount increment + two scalars. Coroutines that need the data capture the `Blob` by value.
- **Automatic lifetime**: the underlying buffer is freed when the last `Blob` copy is destroyed.
- **Mutable → immutable transition**: `freeze()` is a logical transition (not enforced at compile time). Debug builds could assert `m_frozen` in the copy constructor to catch misuse. Once frozen, multiple coroutines can read `data()` concurrently.

#### Public API change (`imager/include/imager/Imager.h`)

```cpp
// Before:
AddResult addImage(const uint8_t* data, size_t size, const std::string& filename);

// After:
AddResult addImage(const Blob& blob, const std::string& filename);
```

This is a **breaking public API change**. All callers must be updated:
- `imager/sample/main.cpp`: use `Blob::fromVector(std::move(data))` for file data read into a vector
- `imager/test/ImagerTest.cpp`: use `Blob::fromVector(std::move(testData))` for test data
- Future HTTP layer: construct `Blob(bodySize)`, write into `writableData()`, `freeze()`

#### Why `shared_ptr<const uint8_t[]>` and not `shared_ptr<char[]>`

The codebase consistently uses `uint8_t` for binary data (validators, hasher, FileStorage). Using `char` would require casts everywhere. `uint8_t` is the natural type for raw binary buffers in C++.

#### Why not `std::span` or `std::string_view`

These are non-owning views — they don't solve the lifetime problem. The whole point is that `Blob` *owns* the data so coroutines can safely hold a copy.

#### How `Blob` flows through the pipeline

**addImage path** (caller provides data):
```
Caller: Blob blob = Blob::fromVector(std::move(fileData));   // frozen
  │
  ├─ addImage(blob, filename)
  │    ├─ validate coroutine captures `blob` by value (refcount++)
  │    ├─ hash coroutine captures `blob` by value (refcount++)
  │    ├─ both use blob.data() / blob.size() concurrently (safe, frozen)
  │    ├─ writeFileAsync captures `blob` by value (refcount++)
  │    │    ├─ per-root coroutines each capture `blob` by value
  │    │    └─ all write blob.data() to disk concurrently
  │    └─ blob goes out of scope → refcount drops to 0 → freed
```

**readFile path** (FileStorage reads from disk into Blob):
```
FileStorage::readFile(id, ext):
  ├─ stat file to get size
  ├─ Blob blob(fileSize);                          // allocate writable
  ├─ stream.read(blob.writableData(), fileSize);    // fill directly
  ├─ blob.freeze();                                 // now immutable
  └─ return blob;                                   // no extra copy
```

This replaces the current `std::vector<uint8_t>` return type of `readFile`, eliminating a copy when the data is subsequently shared with coroutines.

#### `readFile` return type change

```cpp
// Before:
std::vector<uint8_t> readFile(const std::string& id, const std::string& ext);

// After:
Blob readFile(const std::string& id, const std::string& ext);
```

And correspondingly in `Imager`:

```cpp
// Before:
std::vector<uint8_t> getImageData(const std::string& id);

// After:
Blob getImageData(const std::string& id);
```

#### Internal API updates

`FileStorage` methods also change to accept `Blob`:

```cpp
void writeFile(const std::string& id, const std::string& ext, const Blob& blob);
coro::Task<void> writeFileAsync(const std::string& id, const std::string& ext, Blob blob);
```

Note: `writeFileAsync` takes `Blob` **by value** (not by reference) so the coroutine owns a copy that keeps the data alive across suspension points. The synchronous `writeFile` can take by `const&` since it blocks.

`computeSha256` (Hasher) also changes:

```cpp
// Before:
std::string computeSha256(const uint8_t* data, size_t size);

// After:
std::string computeSha256(const Blob& blob);
```

`IValidator::validate` stays unchanged (`const uint8_t* data, size_t size`) since it's an interface implemented by external validators. The caller extracts `blob.data()` and `blob.size()` at the call site.

### 1. FileStorage becomes coroutine-aware

`FileStorage` gains an internal dependency on the thread pool (passed at construction or via method parameter) and exposes coroutine-based versions of its methods for internal use.

#### Approach A (chosen): pass ThreadPool reference, add coroutine methods

```cpp
class FileStorage {
public:
    explicit FileStorage(std::vector<std::filesystem::path> roots,
                         coro::ThreadPool& pool);

    // Existing synchronous API (now implemented via blockOn internally)
    void writeFile(const std::string& id, const std::string& ext,
                   const Blob& blob);
    Blob readFile(const std::string& id, const std::string& ext);
    void deleteFile(const std::string& id, const std::string& ext);

    // Coroutine API for internal use by Imager
    // Takes Blob BY VALUE — each coroutine owns a shared ref to the data.
    coro::Task<void> writeFileAsync(const std::string& id, const std::string& ext,
                                     Blob blob);
    coro::Task<void> deleteFileAsync(const std::string& id, const std::string& ext);

private:
    std::vector<std::filesystem::path> m_roots;
    coro::ThreadPool& m_pool;

    // Write to a single root (coroutine, runs on pool thread)
    // Captures Blob by value for lifetime safety across suspension.
    coro::Task<void> writeToRoot(const std::filesystem::path& root,
                                  const std::string& id, const std::string& ext,
                                  Blob blob);
};
```

#### `writeFileAsync` implementation

```cpp
coro::Task<void> FileStorage::writeFileAsync(const std::string& id,
                                              const std::string& ext,
                                              Blob blob) {
    // blob is captured by value — each per-root coroutine gets its own
    // shared_ptr copy, keeping the underlying buffer alive.
    std::vector<coro::Task<void>> tasks;
    tasks.reserve(m_roots.size());
    for (const auto& root : m_roots) {
        tasks.push_back(writeToRoot(root, id, ext, blob)); // blob copied (refcount++)
    }

    // Await all — if any fail, we need to clean up
    auto results = co_await coro::whenAllSettled(std::move(tasks));

    // Check for failures; clean up successful writes if any failed
    std::exception_ptr firstError;
    std::vector<size_t> succeededIndices;
    for (size_t i = 0; i < results.size(); ++i) {
        if (results[i]) {
            if (!firstError) firstError = results[i];
        } else {
            succeededIndices.push_back(i);
        }
    }

    if (firstError) {
        // Roll back: delete from succeeded roots in parallel
        std::vector<coro::Task<void>> cleanups;
        for (size_t idx : succeededIndices) {
            cleanups.push_back([](coro::ThreadPool& pool,
                                   std::filesystem::path path) -> coro::Task<void> {
                co_await pool.schedule();
                std::error_code ec;
                std::filesystem::remove(path, ec);
            }(m_pool, filePath(m_roots[idx], id, ext)));
        }
        co_await coro::whenAll(std::move(cleanups)); // best-effort
        std::rethrow_exception(firstError);
    }
}

coro::Task<void> FileStorage::writeToRoot(const std::filesystem::path& root,
                                           const std::string& id,
                                           const std::string& ext,
                                           Blob blob) {
    co_await m_pool.schedule(); // resume on worker thread
    std::filesystem::path path = filePath(root, id, ext);
    std::filesystem::create_directories(path.parent_path());
    std::ofstream out(path, std::ios::binary | std::ios::trunc);
    if (!out) throw std::runtime_error("Cannot open for writing: " + path.string());
    out.write(reinterpret_cast<const char*>(blob.data()),
              static_cast<std::streamsize>(blob.size()));
    if (!out) throw std::runtime_error("Write failed: " + path.string());
    // blob goes out of scope here → refcount-- (freed if last copy)
}
```

#### `deleteFileAsync` implementation

Same pattern: one coroutine per root, `whenAll`, ignore individual errors (best-effort).

#### Synchronous methods become thin wrappers

```cpp
void FileStorage::writeFile(const std::string& id, const std::string& ext,
                             const Blob& blob) {
    coro::blockOn(m_pool, writeFileAsync(id, ext, blob)); // blob copied into async
}
```

This keeps the existing synchronous API working (for the CLI sample, tests, etc.) while the facade uses the async versions internally.

### 2. New coroutine primitive: `whenAllSettled`

Unlike `whenAll` (which rethrows the first exception), `whenAllSettled` always waits for all tasks to complete and returns a `vector<exception_ptr>` (nullptr for success). This is needed for the write-then-rollback pattern.

File: `imager/src/coro/WhenAll.h` (add alongside existing `whenAll`).

```cpp
namespace imager::coro {

/// Await all tasks. Returns a vector of exception_ptr (nullptr = success).
/// Always waits for ALL tasks, even if some fail.
template <typename T>
Task<std::vector<std::exception_ptr>> whenAllSettled(std::vector<Task<T>> tasks);

// void specialization
Task<std::vector<std::exception_ptr>> whenAllSettled(std::vector<Task<void>> tasks);

} // namespace imager::coro
```

### 3. Imager::addImage — new signature + parallel validation + hashing

#### New signature

```cpp
// Before:
AddResult addImage(const uint8_t* data, size_t size, const std::string& filename);

// After:
AddResult addImage(const Blob& blob, const std::string& filename);
```

The `Blob` is passed by `const&` at the public API boundary. Internally, coroutines capture it **by value** (cheap shared_ptr copy) so the data remains valid across suspension points regardless of what the caller does after `addImage` returns.

#### Parallel validation + hashing

Current sequential steps 2 and 3 (validate format, compute SHA256) become parallel. Both coroutines capture a `Blob` copy — the underlying buffer is shared and immutable.

```cpp
AddResult Imager::addImage(const Blob& blob, const std::string& filename) {
    // 1. Extract extension (trivial, synchronous)
    std::string ext = Impl::lowercaseExt(filename);
    if (ext.empty())
        return {ErrorCode::UnsupportedFormat, "", "Filename has no extension"};

    const auto* validator = m_impl->findValidator(ext);
    if (!validator && !Impl::isVideoExtension(ext))
        return {ErrorCode::UnsupportedFormat, "", "Unsupported format: " + ext};

    // 2+3. Validate and hash IN PARALLEL
    // Both coroutines capture `blob` by value (shared_ptr refcount++)
    auto [valResult, id] = coro::blockOn(m_impl->pool, [&]() -> coro::Task<
            std::pair<std::optional<validation::ValidationResult>, std::string>> {

        auto hashTask = [](coro::ThreadPool& pool,
                           Blob b) -> coro::Task<std::string> {
            co_await pool.schedule();
            co_return computeSha256(b);
        }(m_impl->pool, blob);  // blob copied into coroutine

        std::optional<validation::ValidationResult> valRes;
        if (validator) {
            auto valTask = [](coro::ThreadPool& pool,
                              const validation::IValidator* v,
                              Blob b)
                    -> coro::Task<validation::ValidationResult> {
                co_await pool.schedule();
                co_return v->validate(b.data(), b.size());
            }(m_impl->pool, validator, blob);  // blob copied into coroutine

            // Await both
            auto vr = co_await std::move(valTask);
            valRes = vr;
        }
        auto hashResult = co_await std::move(hashTask);
        co_return std::make_pair(valRes, std::move(hashResult));
    }());

    // Check validation result
    if (valResult && !valResult->valid)
        return {ErrorCode::BrokenFile, "", valResult->errorMessage};

    // 4. Duplicate check (synchronous read on first DB)
    // ... same as before but using m_impl->dbs ...

    // 5. Write to storage — blob copied into async coroutines
    m_impl->storage.writeFileAsync(id, ext, blob);  // blob by value into async

    // 6. Insert into DBs (parallel from 0002)
    m_impl->dbs.addFile(id, filename, blob.size(), ext);

    return {ErrorCode::Ok, id, ""};
}
```

**Optimization for video files**: when there is no validator (video extension), only the hash task is launched. No coroutine overhead for the single-task case — use a direct call instead:

```cpp
if (!validator) {
    // Video: no validation, just hash (no point in coroutine for one task)
    id = computeSha256(blob);
} else {
    // Image: validate + hash in parallel
    // ... coroutine fan-out as above ...
}
```

### 4. Imager query methods — parallel tag fan-out

`getImagesByTags` and `listImages` both suffer from an N+1 pattern: for each file result, a separate `getTagsForFile` query is issued. These per-file tag lookups are independent and can run concurrently.

#### New helper: `Impl::enrichWithTags`

```cpp
coro::Task<std::vector<ImageInfo>> Impl::enrichWithTags(
        std::vector<db::File> files) {
    if (files.empty()) co_return {};

    // Build one coroutine per file to fetch its tags
    std::vector<coro::Task<std::vector<std::string>>> tagTasks;
    tagTasks.reserve(files.size());
    for (const auto& f : files) {
        tagTasks.push_back(
            [](coro::ThreadPool& pool, MultiDatabase& dbs,
               std::string fileId) -> coro::Task<std::vector<std::string>> {
                co_await pool.schedule();
                co_return dbs.getTagsForFile(fileId);
            }(pool, dbs, f.id)
        );
    }

    auto allTags = co_await coro::whenAll(std::move(tagTasks));

    std::vector<ImageInfo> result;
    result.reserve(files.size());
    for (size_t i = 0; i < files.size(); ++i) {
        result.push_back(toImageInfo(files[i], std::move(allTags[i])));
    }
    co_return result;
}
```

#### Updated methods

```cpp
std::vector<ImageInfo> Imager::getImagesByTags(
        const std::vector<std::string>& tags,
        uint32_t offset, uint32_t limit) {
    if (tags.empty()) return {};
    try {
        auto files = m_impl->dbs.getFilesByTags(tags, db::Pagination{offset, limit});
        return coro::blockOn(m_impl->pool, m_impl->enrichWithTags(std::move(files)));
    } catch (const db::DatabaseException&) {
        return {};
    }
}

std::vector<ImageInfo> Imager::listImages(uint32_t offset, uint32_t limit) {
    try {
        auto files = m_impl->dbs.getAllFiles(db::Pagination{offset, limit});
        return coro::blockOn(m_impl->pool, m_impl->enrichWithTags(std::move(files)));
    } catch (const db::DatabaseException&) {
        return {};
    }
}
```

**Note on thread safety for reads**: `MultiDatabase` reads go to `m_dbs[0]`. The `db::Database` read methods take a `shared_lock`, so multiple concurrent `getTagsForFile` calls from different pool threads are safe — they all acquire shared locks and execute concurrently in WAL mode.

### 5. Imager::deleteImage — parallel storage cleanup

After the DB delete succeeds, the storage cleanup across roots can be parallel:

```cpp
ErrorCode Imager::deleteImage(const std::string& id) {
    std::lock_guard<std::mutex> lock(m_impl->writeMutex);

    auto file = m_impl->dbs.getFile(id);
    if (!file) return ErrorCode::FileNotFound;
    const std::string ext = file->ext;

    try {
        m_impl->dbs.deleteFile(id); // parallel across DBs (from 0002)
    } catch (const db::DatabaseException& e) {
        if (e.code() == db::DatabaseErrorCode::NotFound)
            return ErrorCode::FileNotFound;
        return ErrorCode::DatabaseError;
    }

    // Storage cleanup — parallel across roots (best-effort)
    coro::blockOn(m_impl->pool, m_impl->storage.deleteFileAsync(id, ext));
    return ErrorCode::Ok;
}
```

### 6. Imager::Impl — updated structure

```cpp
struct Imager::Impl {
    coro::ThreadPool                                      pool;
    MultiDatabase                                         dbs;
    FileStorage                                           storage;
    std::vector<std::unique_ptr<validation::IValidator>>  validators;
    std::mutex                                            writeMutex;

    explicit Impl(const config::AppConfig& cfg)
        : pool(std::min<size_t>(std::thread::hardware_concurrency(),
                                 cfg.targets.size() * 2))
        , dbs(cfg.targets, pool)   // MultiDatabase now takes pool reference
        , storage(extractRoots(cfg.targets), pool)
        , validators(createDefaultValidators())
    {}

    static std::vector<std::filesystem::path> extractRoots(
            const std::vector<config::TargetConfig>& targets) {
        std::vector<std::filesystem::path> roots;
        roots.reserve(targets.size());
        for (const auto& t : targets) roots.push_back(t.root);
        return roots;
    }

    const validation::IValidator* findValidator(const std::string& ext) const {
        for (const auto& v : validators)
            if (v->supportsExtension(ext)) return v.get();
        return nullptr;
    }

    static bool isVideoExtension(const std::string& ext) {
        return ext == ".mp4" || ext == ".mov";
    }

    static std::string lowercaseExt(const std::string& filename) {
        auto pos = filename.rfind('.');
        if (pos == std::string::npos) return {};
        std::string ext = filename.substr(pos);
        for (auto& c : ext)
            c = static_cast<char>(std::tolower(static_cast<unsigned char>(c)));
        return ext;
    }

    static ImageInfo toImageInfo(const db::File& f,
                                  std::vector<std::string> tags = {}) {
        return ImageInfo{f.id, f.name, f.size, f.ext, std::move(tags)};
    }

    // Coroutine helper: fetch tags for a batch of files in parallel
    coro::Task<std::vector<ImageInfo>> enrichWithTags(std::vector<db::File> files);
};
```

### 7. ThreadPool ownership — single pool, shared by all

The `ThreadPool` is created once in `Imager::Impl` and passed by reference to `MultiDatabase` and `FileStorage`. This avoids multiple pools competing for threads.

**MultiDatabase change** (from 0002): instead of owning its own pool, it receives a `coro::ThreadPool&` at construction:

```cpp
class MultiDatabase {
public:
    explicit MultiDatabase(const std::vector<config::TargetConfig>& targets,
                           coro::ThreadPool& pool);
    // ...
private:
    std::vector<std::unique_ptr<db::Database>> m_dbs;
    coro::ThreadPool& m_pool;  // reference, not owned
};
```

### 8. Thread pool sizing

The pool serves three kinds of work:
- **DB operations**: N databases, typically 2–4
- **Storage I/O**: N roots (same count as databases after 0002)
- **CPU work**: validation, hashing (at most 2 concurrent tasks)

A reasonable default: `max(4, hardware_concurrency)`, capped at 16 to avoid over-subscription. Since all workloads are I/O-bound (SQLite, filesystem) except validation/hashing, a moderate pool is sufficient.

```cpp
static size_t defaultPoolSize(size_t numTargets) {
    size_t hw = std::thread::hardware_concurrency();
    if (hw == 0) hw = 4;
    return std::clamp(hw, std::max<size_t>(4, numTargets), size_t{16});
}
```

## Public API impact

**Breaking change to `addImage` signature.** The method changes from:

```cpp
AddResult addImage(const uint8_t* data, size_t size, const std::string& filename);
```

to:

```cpp
AddResult addImage(const Blob& blob, const std::string& filename);
```

All callers must be updated. The `Blob` class lives in `imager/include/imager/types/Blob.h` and is re-exported via `Types.h`.

Additionally, `getImageData` return type changes from `std::vector<uint8_t>` to `Blob`:

```cpp
// Before:
std::vector<uint8_t> getImageData(const std::string& id);

// After:
Blob getImageData(const std::string& id);
```

The rest of the public API (`getImage`, `deleteImage`, `tagImage`, etc.) is unchanged.

## Risks and mitigations

- **Data lifetime (eliminated by `Blob`)**: previously, raw `uint8_t*` pointers passed to `addImage` had to remain valid for the duration of all coroutines. With `Blob`, each coroutine captures a copy (shared_ptr refcount bump), guaranteeing the underlying buffer survives all suspension points. This also makes the API safe for a future non-blocking `addImage` variant.

- **`Blob` construction cost**: `Blob::fromVector` copies the vector contents into a `shared_ptr`-managed array (one `memcpy`). The `Blob(size)` constructor allocates without copying — callers fill the buffer directly via `writableData()`. For file reading (`FileStorage::readFile`), this means zero extra copies: allocate → read from disk into `writableData()` → `freeze()`. All subsequent copies within the pipeline are O(1) shared_ptr refcount bumps.

- **ThreadPool starvation**: if all pool threads are blocked on DB operations (e.g., SQLite busy_timeout of 5 seconds), new coroutines queue up. This is acceptable for the expected scale (2–4 targets). The pool size is set to be >= the number of targets, so at least one full parallel fan-out can proceed without waiting.

- **Exception safety**: `whenAllSettled` ensures all tasks complete even on failure, preventing orphan coroutines. Rollback in `writeFileAsync` is best-effort — if a cleanup fails, the partially-written file is left behind (same as the current sequential behavior).

- **Read contention on `m_dbs[0]`**: the parallel `enrichWithTags` sends N concurrent `getTagsForFile` calls to the first database. Since `db::Database` uses `shared_lock` for reads and SQLite WAL mode supports concurrent readers, this is safe. If the page limit is 50, that means up to 50 concurrent shared locks — SQLite handles this without issue.

- **blockOn deadlock risk**: if a coroutine awaiting on the pool calls `blockOn` on the same pool, and all pool threads are occupied, deadlock can occur. This is prevented by design: `blockOn` is only called from the main (caller) thread in `Imager` methods, never from within a pool coroutine. The coroutines themselves use `co_await`, not `blockOn`.

## Parallelism summary by operation

| Operation | Before | After |
|-----------|--------|-------|
| `addImage` validation + hash | Sequential: V then H | Parallel: V ∥ H |
| `addImage` storage write | Sequential: root₁ then root₂ ... | Parallel: root₁ ∥ root₂ ∥ ... |
| `addImage` DB insert | Sequential (0002 makes parallel) | Parallel: db₁ ∥ db₂ ∥ ... |
| `deleteImage` DB delete | Parallel (from 0002) | Parallel: db₁ ∥ db₂ ∥ ... |
| `deleteImage` storage delete | Sequential: root₁ then root₂ ... | Parallel: root₁ ∥ root₂ ∥ ... |
| `getImagesByTags` tag fetch | N+1 sequential loop | Parallel: tag₁ ∥ tag₂ ∥ ... tagₙ |
| `listImages` tag fetch | N+1 sequential loop | Parallel: tag₁ ∥ tag₂ ∥ ... tagₙ |
| `tagImage` | Single DB call (parallel from 0002) | Parallel: db₁ ∥ db₂ ∥ ... |
| `untagImage` | Single DB call (parallel from 0002) | Parallel: db₁ ∥ db₂ ∥ ... |
| `createTag` | Single DB call (parallel from 0002) | Parallel: db₁ ∥ db₂ ∥ ... |
| `deleteTag` | Single DB call (parallel from 0002) | Parallel: db₁ ∥ db₂ ∥ ... |
| `getImage` | 2 sequential calls | Sequential (not worth parallelizing) |
| `getImageData` | DB read → storage read | Sequential (data dependency) |
| `imageCount` | Single DB read | Sequential (single call) |
